# Yueliang 架构设计

## 核心设计哲学

**「内部消化，仅输出音频」**

彻底规避传统 DAW 处理海量 MIDI 事件时的性能瓶颈。

---

## 四大核心机制

### 1. 宿主隔离 (DAW Isolation)

- 插件作为普通 VSTi 挂载在 DAW 中
- **绝不**将巨量 MIDI Note 数据发送到 DAW 钢琴卷帘窗
- 通过 `nih-plug` 实时获取 DAW 走带位置
- 内部维持与 DAW 同步的时间轴

### 2. 内部合成与渲染 (Internal Synthesis)

- 不使用 `xsynth-realtime`（不依赖物理声卡后端）
- 在 `nih-plug` 音频处理线程中直接调用 `xsynth-core`
- 按块（Block）渲染，将 `f32` 采样数据填充到 DAW 输出缓冲区
- **无缝离线渲染**：完全受 DAW 时钟调度，导出时以最快运算速度生成音频

### 3. 硬核参数与预过滤 (Pre-filtering)

- 暴露给 DAW 的参数极度精简，专为黑乐谱定制
- 动态拦截机制：音符喂给 xsynth 前进行事件拦截
- `Velocity Threshold`：直接丢弃低于阈值的「幽灵音符」

### 4. 多通道路由 (Multi-channel Routing)

- 向 DAW 申请多个音频输出总线
- 提供内部路由矩阵
- 将 MIDI 通道动态映射到不同 VST 输出总线

## 关键技术决策

### 为什么不用 xsynth-realtime?

xsynth-realtime 使用 CPAL 等物理声卡后端，适合独立应用程序。  
Yueliang 作为 VST 插件，音频输出由 DAW 控制，直接在 `process()` 中渲染即可。

### 为什么 MIDI_INPUT = None?

黑乐谱的音符数量巨大，走 DAW 的 MIDI 输入会导致：
1. DAW 钢琴卷帘窗卡顿
2. 其他音源被迫播放海量音符

解决方案：内部加载 MIDI 文件，DAW 只提供 Transport 时钟。
