# Yueliang 参数设计原理

## 问题

为什么暴露 Gain、Max Polyphony 等参数给 DAW，而不是 CC（Control Change）？

## 答案：分层控制架构

### 两个控制层级

| 层级 | 数据类型 | 来源 | 精度 | 用途 |
|------|----------|------|------|------|
| **音符层** | MIDI Note/CC/Pitch Bend | 内部序列器 | 采样级 | 微观演奏细节 |
| **控制层** | Gain/Polyphony/Filter | DAW 自动化 | 缓冲区级 | 宏观混音控制 |

### 数据流对比

```
传统 VSTi：
DAW MIDI轨 → [CC #7 音量] → 插件渲染

Yueliang（DAW 隔离）：
内部黑乐谱 → [CC/音符] → XSynth 渲染
                    ↑
DAW 自动化 → [Gain] → 最终音量调节
```

## 为什么这样设计？

### 1. DAW 隔离（核心架构）

- DAW 钢琴卷帘**无法承载上亿音符**
- 内部序列器预加载黑乐谱数据到内存
- DAW 只负责走带（播放/暂停/跳转）和接收音频输出

### 2. 性能优化

- **内部处理**：采样级精度，处理每个音符事件
- **DAW 控制**：缓冲区级精度足够（每 64-1024 采样更新一次）

### 3. 职责分离

| 组件 | 角色 | 控制范围 |
|------|------|----------|
| DAW | 录音室调音台 | 全局音量、效果器、走带 |
| Yueliang | 乐手 + 乐器 | 演奏细节、音色、复音管理 |

## 当前参数用途

| 参数 | 类型 | 用途 |
|------|------|------|
| `gain` | Float | 总输出音量（dB） |
| `max_polyphony` | Int | 性能限制，防止爆音（1-100000） |
| `velocity_threshold` | Int | 黑乐谱力度过滤（0-127） |
| `force_max_velocity` | Bool | 强制最大力度开关 |
| `interpolation` | Enum | 采样插值质量（Nearest/Linear） |
| `enable_limiter` | Bool | 输出限幅器开关 |

## 等 XSynth 集成后

CC **仍然不会**暴露给 DAW：

| CC | 传统用途 | Yueliang 处理方式 |
|----|----------|-------------------|
| CC #7 | 通道音量 | 由 Gain 参数替代 |
| CC #10 | 声像 | 由内部多通路路由处理 |
| CC #64 | 延音踏板 | 由音符事件内部携带 |
| CC #1 | 调制轮 | 由内部序列器生成 |

## 总结

> **DAW 控制的是「录音室层面的调节」，而「演奏层面的细节」完全在插件内部处理。**

这种设计使得：
1. DAW 界面保持简洁（不显示亿级音符）
2. 内部可以精确处理黑乐谱的复杂演奏
3. 离线渲染时可以任意跳转时间线
